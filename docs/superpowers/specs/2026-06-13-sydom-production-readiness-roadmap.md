# 司域 (Sydom) 生产就绪路线图 — 差距分析与里程碑计划

> 类型：**项目群路线图（program roadmap）**，非单功能 spec。本文负责「全面梳理差距 → 排优先级 → 排里程碑」，每个里程碑后续各自走独立的 brainstorm → spec → plan → 实现周期。
> 日期：2026-06-13

## 1. 背景与目标

**现状（MVP 已完成）**：司域已端到端可运行——数据面（控制面策略核心 → outbox/relay → Sidecar 内核 + 数据权限引擎 → 公开 SDK）+ 管理/接入面（gRPC AdminService 27 RPC + REST 网关 + Web Console 服务端 BFF）。端到端 Demo 实证功能权限 / 数据权限 / fail-close。详细设计与接入面大特性（SP1/SP2/SP3）均已并入 `main`（`82390d4`）。

**目标**：从「功能 MVP」迈向**生产就绪的对外商业 / SaaS 产品**。「生产就绪」在此不仅指可靠运行，更包含**功能完整性**与**产品体验**——尤其要让**不懂技术的业务人员看懂会用**。

**本轮 brainstorm 锁定的产品决策**：

| 维度 | 决策 | 含义 |
|---|---|---|
| 目标形态 | 对外商业 / SaaS 产品 | 多租户、功能完整性、产品体验均为第一公民 |
| 租户模型 | **全托管多租户（共享控制面）** | 隔离边界成架构级 P0；企业专享部署 **deferred**（有客户明确要求再做） |
| 产品体验 | **双层 UX**：业务向「运营台」+ 技术向「建模台」 | 运营台面向不懂技术的业务人员；建模台面向客户的开发/安全团队。两套界面、两种语言 |
| 排序策略 | **A（纵向可演示切片）为主干 + B/C 纪律** | 每里程碑可演示可给设计客户用；多租户基座 M1 一次做扎实；两条存在性风险早验证 |

**贯穿原则（从既有工程范式继承）**：一致性优先于简化（fail-close / DB 真相源 / 缓存全量清）；三方（及未来运营台/建模台）共用 `AuthorizeRule` / `CheckStatusWrite` / `ruleTable` 一份授权真相，绝无旁路；casbin 行为论断先回源核实；子代理驱动 + 两阶段审查 + 整体安全评审。

## 2. 差距全景（基于 27-RPC surface + Console + DB schema 实查，非臆测）

### A. 多租户与账户体系（SaaS 基座 — 全新）
- **租户实体与隔离边界**：现模型按「app-域」（domain = 数字 app_id）隔离，`tenant` 表存在但无 tenant 级隔离；需上提一层 tenant 边界——租户拥有多 app、租户管理员只见本租户、跨租户在物理上不可能。
- **身份分层**：现 `root` 是全局超管（域 `*`），不区分「SaaS 运营方」与「客户租户管理员」；需引入租户管理员（owner/admin/member）且其权力天花板锚定在本租户。
- **客户自助**：注册 / 邀请成员 / 租户内角色；当前无任何自助账户能力。
- **商业化基座**：计费 / 配额 / 套餐 / 用量计量；企业 SSO（SAML/OIDC）。

### B. 功能完整性（授权产品内核 — 实查缺口）
- **决策可解释 explain**：「为什么 allow / deny」——casbin 支持 explain，AdminService 未暴露任何 explain RPC。
- **有效权限 / 反查**：「user X 到底能做什么」「谁能做动作 Y / 谁有权限 Z」——无任何 RPC（仅有正向 List）。
- **审计查询**：`policy_audit_log` 在库（且 `diff` 列恒 NULL）但**无读取 RPC**；操作员看不到变更历史 / 谁改的。
- **生命周期不完整**：Role / Application **无改名（Update）**；Application / Operator / AdminRole **无删除**；无 Get-single。
- **撤权不对称**：`GrantAdminRole` / `BindOperatorRole` **只能授不能撤**（无 RevokeAdminGrant / UnbindOperatorRole）——能给权不能收权，是一致性 / 安全缺口。
- **Secret 无轮换 / 重置**：Application / Operator secret 一次性签发后无重置路径——泄露无补救。
- **零分页 / 搜索 / 排序 / 过滤**：所有 List RPC 与 Console 列表均无界 `ORDER BY id`——规模化下 API 与界面均不可用。
- **批量 / 导入导出 / 策略即代码（IaC）**：无。
- **保存前决策模拟**：无「预演这条策略会让谁能做什么」。

### C. 产品体验（双层 UX + 业务语言抽象层 — 实查缺口）
**核心支柱：业务语言抽象层**。现 Console 直接暴露 casbin/RBAC 原语（ptype、domain、resource/action、condition JSON、role_id 裸整数、"operator"/"admin-role" 黑话），不懂技术的业务人员无从下手。需把原语翻译成业务概念（"谁能看哪些订单"、"给销售经理开通导出"）。
- **业务向运营台**：把人分配到业务角色、看「谁能做什么」、申请 / 审批、自助——全业务语言。
- **技术向建模台**：定义资源 / 动作 / 条件、SDK 接入、IaC——面向开发 / 安全团队。
- **信息架构与基础体验**：概览仪表盘（计数 / 最近变更 / 健康）、面包屑、空状态；危险操作二次确认、表单校验反馈、成功 / 失败 toast（现仅 PRG 静默）。
- **列表能力**：搜索 / 过滤 / 分页 UI、排序、批量选择、内联编辑。
- **关系可视化**：角色继承图、有效权限浏览器、决策模拟器。
- **打磨与可及性**：设计系统 / 视觉打磨、响应式、深色模式、i18n、无障碍（a11y）。
- **引导与 DX**：自助 onboarding 向导、in-product 帮助；开发者侧 API 文档门户、SDK quickstart、密钥管理界面、sandbox 测试模式。

### D. 运维就绪
- **可观测性**：结构化日志（现薄、多 TODO）、metrics（无）、tracing（无）、健康 / 就绪探针（无）。
- **高可用**：控制面多实例故事未定、relay 单点、PG / Redis HA、故障转移。
- **安全硬化**：全链 TLS / mTLS、登录限流 / 防爆破、会话策略、密钥轮换、依赖扫描。（注：`handleLogout` 不校验 CSRF 已经整体评审裁定保持现状。）
- **部署**：helm / k8s、配置管理、迁移自动化（现假定预迁移）、零停机、备份恢复。
- **性能**：压测 / 容量规划、缓存校验、连接池调优。
- **合规**：审计留存、数据驻留、GDPR / 删除权、SLA / SLO。

### E. 质量与可信
- **测试**：负载 / 浸泡、混沌（分区下 fail-close）、安全渗透。
- **文档**：运维 runbook、API 参考、SDK 文档、采用者架构文档。
- **发布工程**：API 版本化、向后兼容策略、变更管理。

## 3. 排序策略与理由

工作量分 7 条工作流：①多租户基座 ②功能完整性 ③业务向运营台 ④技术向建模台 + 开发者 DX ⑤运维硬化 ⑥商业化（计费 / 配额 / SSO）⑦质量可信。

**采用策略 A（纵向可演示切片）为主干，吸收 B（地基优先）与 C（风险优先）的纪律**：
- **A**：每个里程碑产出可演示、可给设计客户用的完整增量，学习闭环最快、最早验证 PMF；契合本项目已验证的增量交付范式（e2e Demo、SP1/2/3 每个都是可运行增量）。
- **B 纪律**：多租户基座在 M1 一次做扎实——隔离边界事后改造极痛，是唯一必须「地基优先」的部分。
- **C 纪律**：两条存在性风险——**租户隔离正确性**（安全致命）与**业务 UX 可用性**（产品致命）——在 M1 就拿到信号、M3 充分验证，不留到后面。

被否方案：纯 B（横向分层）——很久才有可卖的东西、做错方向风险最高、无早期反馈；纯 C（风险优先）——不够可演示、里程碑不好切。

## 4. 里程碑路线图

> 时间轴直觉：**M1–M3 = 让设计客户用起来并验证 PMF**（私有 Beta，可免费 / 手动计费）；**M4–M6 = 规模化与商业化到 GA**。每个里程碑都穿一层「最小可托管」运维底座（不把全部运维堆到 M5）。

### M1 · 多租户基座 + 端到端最薄业务旅程 → 首个私有 Beta
- **范围**：租户实体 + 隔离边界（租户拥有多 app；「运营方 root ↔ 租户管理员」分层；读写全 tenant-scoped；跨租户 403，安全矩阵扩到租户级）；自助账户最小集（租户注册 + 首个管理员 + 邀请成员）；最小功能（把人分配到角色 + **「某人能做什么」有效权限视图**）；一条业务语言薄运营台旅程；技术建模沿用现 Console；含「最小可托管」运维底座（TLS、健康探针、可部署）。
- **退风险**：租户隔离正确性 ✅ + 业务 UX 可用性首个信号 ✅。
- **验收**：跨租户 403 安全矩阵全绿；非技术人能无辅助完成「把 Alice 设为销售经理并看她能做什么」。

### M2 · 授权产品功能纵深（operate / understand）
- **范围**：explain「为什么 allow / deny」；反查「谁能做 Y」；审计查询 API + 变更历史；生命周期补全（改名 / 删除 / Get-single）；**撤权对称**（RevokeAdminGrant / UnbindOperatorRole）；**Secret 轮换 / 重置**；全 List 分页 / 搜索 / 排序 / 过滤。
- **验收**：每资源全 CRUD 生命周期；「为什么 / 谁能 / 谁改了」均可答；所有列表分页。

### M3 · 业务向运营台成体系（业务语言抽象层 + 体验打磨）
- **范围**：原语 → 业务概念翻译层 + 预设 / 模板；关系可视化（继承图 / 有效权限浏览器 / **决策模拟器**）；体验打磨（仪表盘 / 二次确认 / 校验反馈 / toast / 空状态 / 面包屑 / 搜索过滤分页 UI / 批量）；设计系统 / 响应式 / 深色 / i18n / a11y；自助 onboarding 向导 + in-product 帮助。
- **退风险**：业务 UX 可用性随真实设计客户充分验证 ✅。
- **验收**：真实非技术用户可用性测试达标（任务完成率 / SUS 分）。

### M4 · 技术向建模台 + 开发者 DX
- **范围**：建模台重整（资源 / 动作 / 条件构建器增强、策略即代码 / IaC、导入导出、批量）；开发者 DX（API 文档门户、SDK quickstart、密钥管理界面、sandbox 测试模式）。
- **验收**：首次集成耗时指标；API 文档发布。

### M5 · 运维就绪 + 生产硬化
- **范围**：可观测性（结构化日志 / Prometheus metrics / tracing / 健康就绪探针）；HA（控制面多实例 / relay 去单点 / PG·Redis HA / 优雅故障转移）；安全硬化（全链 TLS·mTLS / 登录限流防爆破 / 会话策略 / 密钥轮换 / 依赖扫描）；部署（helm·k8s / 迁移自动化 / 零停机 / 备份恢复）；性能（压测 / 容量规划 / 缓存校验）。
- **验收**：SLO 达标；压测通过；混沌（分区下 fail-close）绿；安全扫描干净。

### M6 · 商业化 + 合规 + GA
- **范围**：计费 / 配额 / 套餐 / 用量计量；企业 SSO（SAML/OIDC）；合规（审计留存 / 数据驻留 / GDPR·删除权 / SLA·SLO）；渗透测试；运维 runbook + 完整文档；API 版本化与向后兼容策略。
- **验收**：渗透测试通过；计费 E2E；合规清单；GA 文档齐备。

## 5. 跨里程碑的架构与一致性约束（carry-forward）

- **一份授权真相**：运营台、建模台、REST、gRPC 全部经导出的 `AuthorizeRule` / `CheckStatusWrite` / 唯一 `ruleTable`；多租户的 tenant 边界须并入同一鉴权核心（**不得**在运营台另起一套租户判定逻辑）。
- **fail-close 全贯穿**：租户隔离、有效权限计算、决策模拟、HA 分区下，宁可拒绝 / 报错，绝不放行 / 泄露。
- **DB 真相源**：tenant / 账户 / 计费状态均以 DB 为准，缓存变更全量清。
- **回源核实**：explain / 有效权限 / 反查若依赖 casbin 能力（GetImplicitPermissions 等），实现前读 v3.10.0 源码核实语义。
- **范式延续**：每里程碑（或其子项目）走子代理驱动 + 两阶段审查 + 整体安全评审；跨包改签名后 `go vet ./...` 全仓兜底。

## 6. 假设与未决项

- **近期里程碑 bar 假设**：M1–M3 = 设计客户私有 Beta（免费 / 手动计费），M6 = GA。若你的近期目标不同（如直接公开 GA、或仅内部试点），里程碑 bar 需相应调整。
- **企业专享 / 单租户部署**：deferred，留到有 enterprise 客户明确要求时作为增量子项目。
- **计费 / SSO / 设计系统**具体选型：留到对应里程碑各自 brainstorm。
- **租户隔离的实现路线**（在现 app-域模型上加 tenant 维度 vs 重构域模型）将在 M1 的 brainstorm/spec 中定夺——这是 M1 最大的架构决策点。
