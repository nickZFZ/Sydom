## M4.2 批量操作（Bulk Operations）+ 三面 parity（子代理驱动，FF 并 push origin，2026-07-05）

## M5.5 BatchEnforce 决策缓存复用（子代理驱动执行 plan，FF 入 main 并 **已 push origin，origin/main==本地 main==`d18f134`**，2026-07-16）

补 M5.5 基准发现的批量路径未复用缓存缺口（旧 BatchEnforce ~2.13ms/50=~42µs/决策 vs 单条缓存命中 137ns）。**方案**：`kernel.BatchEnforce` 从调 casbin `e.ce.BatchEnforce`（经嵌入 SyncedEnforcer.BatchEnforce〔持批次级 RLock〕→Enforcer.BatchEnforce 直调底层 `enforce("",nil,..)` 绕决策缓存）**改为逐行调 `e.ce.Enforce`**（SyncedCachedEnforcer 的缓存 Enforce）。**零新增缓存逻辑**——复用 casbin 自身 get/set/键推导/失效。**关键事实（回源核实 casbin v3.10.0）**：SyncedCachedEnforcer 只覆写 Enforce 不覆写 BatchEnforce；单条与批量最终同走 `e.enforce("",nil,..)`、matcher 同为 `""`，`GetCacheKey` 对全 string 参数逐字相同→**批量与单条共享缓存键、互相暖缓存**。**实测（同机 Ryzen9 7900X/Go1.26.3）**：暖缓存批量 2,130,430→**7,522 ns/op（~283×）**≈150ns/决策≈单条命中速度；**冷/全新元组批量仍 ~2.13ms**（收益仅暖缓存兑现，已在手册诚实标注）。**取舍**：放弃批次原子性（逐行各自取放 RLock→一批可跨相邻两版本撕裂）；论证=BatchCheck 语义≡N×Check（客户端发 N 次单条本就撕裂）、原子性从非承诺契约、撤权及时性靠 apply 的 `InvalidateCache` 全量清（顺序流下后续探测必 miss 重算，T4 钉死）；并发 set-after-clear 窄窗与单条 Check 同款、非本片新增。**footprint 机器 diff 核验=恰 3 文件**（engine.go〔改动全落 BatchEnforce 函数体+注释,`grep '^[+-]func'` 空证无其它函数触碰,GetImplicitRolesForUser 体逐字不变〕/engine_test.go/performance-baselines.md）；cache.go/authz/effperm 逐字未改。**TDD 双证结构（plan 关键洞察）**：只 T1 是 RED→GREEN（注入 countingCache 计 Get 命中,现行批量绕缓存故 hits=0 红）；T2/T3/T5 契约刻画（现行即绿、改后仍绿=行为不变）+T4 撤权一致性（批量孪生）对现行即绿——**牙来自变异非初始红**；故序=先 T2-T5→T1 红→改实现→**变异验证延到实现改写后**（时序陷阱:摘 InvalidateCache 在改写前无感,朴素 TDD 会得"变异仍绿"假象险误删最重要测试）。两变异均经终审独立复现:摘 InvalidateCache→T4 红、还原 BatchEnforce→T1 红。**执行=子代理驱动**（7 任务各 implementer+spec 审+quality 审;auth-core 改动 opus 深审+`-race`;opus 终审整体"可合并"）。**踩坑**:`git commit --amend` 被沙箱权限拦（子代理与我均拦）→改用普通 follow-up 提交（d25dfd4/d18f134 两处注释精度收敛,均纯注释 go vet+7 BatchEnforce 测试 -count=1 绿）。7 提交 `be79f96..d18f134`,`go test ./...` EXIT 0（41 ok 包）。**M5.5 债剩:压测/CI benchstat（本片只改批量热路径,未做端到端压测/CI 回归门）。**
